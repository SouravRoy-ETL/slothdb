Community

SlothDB is an independent open-source project. It's MIT licensed, built in the open, and shaped by the people who use it. Here's how to reach out.

Get help

Issues

Bug reports, crashes, wrong results, and feature requests. The fastest way to get a fix: include a minimal reproduction, the SQL you ran, and the input file (or a synthetic version of it).

Open an issue on GitHub ->

Discussions

Questions, design discussion, benchmark debates, and "how do I do X" without it being a bug. Also where new features get scoped before they become issues.

Browse discussions ->

Contact the author

For anything that does not fit in a public issue (licensing, commercial use, partnership), reach out on GitHub. Open a direct thread via the profile, or start a private discussion.

Security

Please disclose security issues privately first. Bounds-checked parsing and DoS limits are in place, but the project is young. Report via a private GitHub security advisory and we will coordinate a fix before public disclosure.

Contribute

Pull requests are welcome for bug fixes, missing SQL features, new file format readers, docs, and benchmark improvements.

First-time contributors

Good entry points: failing cases in the test suite, missing SQL functions from the standard dialect, docs improvements, new examples. Look for issues tagged good-first-issue.

Build + test

git clone https://github.com/SouravRoy-ETL/slothdb
cd slothdb
cmake -B build -DSLOTHDB_BUILD_SHELL=ON \
  -DSLOTHDB_BUILD_TESTS=ON
cmake --build build --config Release
ctest --test-dir build -C Release

Coding standards

C++20, .clang-format enforced. Avoid adding dependencies. Every perf optimization should ship with a benchmark delta in the commit message.

Testing

403 tests, 131 000+ assertions via doctest. Every new feature needs a test. Benchmarks live in real-life-testing/ (run before and after perf changes).

Project status

SlothDB is v0.1.7. About seven months old, one primary author, CI-green on Windows, Linux and macOS. Use it in real workflows, expect to file bugs. It's not production-critical yet.

The project is developed in the open. Every optimization, every benchmark change, every extension is one commit in the git log.

Release history

  • v0.1.7: interactive .ask natural-language-to-SQL sub-REPL in the CLI shell (rules-first, ~50 KB; optional local Qwen GGUF fallback under -DSLOTHDB_ASK_MODEL=ON), CREATE TABLE AS SELECT (including from files), catalog-introspection C API for Python / Node / WASM bindings, README integrity pass (retracted the "1.1× to 8.6×" range; honest range is 1.04× to 5.43×, median 1.70×)
  • v0.1.6: JOIN hot-path overhaul (288 ms -> 85 ms on big × small), CREATE LIVE VIEW with incremental CSV-append path for log-tail workloads, SLOTHDB_EDGE build for sub-MB WASM bundles (Cloudflare Workers target), DESCRIBE, PRAGMA table_info / database_list, VARCHAR(n) length enforcement, ORDER BY on narrowed projections, JOIN reverse-order equi-join fix
  • v0.1.5: WASM playground at slothdb.org/playground, @slothdb/wasm npm package, binder / planner fixes (non-reserved keywords as identifiers, SELECT * ORDER BY, UNION + JOIN file-literal preprocessing, COPY with bare file literal)
  • v0.1.4: remote file reading (https:// and public s3:// URLs), extended DATE_TRUNC (week/quarter/decade/etc.), new scalars MONTHNAME / DAYNAME / LAST_DAY / MAKE_DATE
  • v0.1.3: Arrow IPC and SQLite on the fast path, PhysicalArrowScan and PhysicalSQLiteScan
  • v0.1.2: CSV beats DuckDB on every query (vectorized filter, fused WHERE, parallel no-GROUP-BY aggregate)
  • v0.1.1: JSON parser rewrite, Excel streaming scan
  • v0.1.0: first public release with CSV, Parquet, JSON, CLI shell, Python, C/C++ API

See the full CHANGELOG for per-commit details.
